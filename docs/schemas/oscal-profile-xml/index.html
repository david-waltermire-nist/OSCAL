OSCAL Profile Model - XML Schema Reference

Schema download oscal_profile_schema.xsd

Schema version 1.0.0-milestone2

Short name oscal-profile

XML namespace http://csrc.nist.gov/ns/oscal/1.0

A profile designates a selection and configuration of controls from one or more catalogs, along with a series of operations over them. The topmost element in the OSCAL profile XML schema is profile.

Addition: add element

Specifies contents to be added into controls, in resolution

This element appears inside: alter.

The add element has the following contents (in order):

  • position attribute (optional), type string

    The value must be one of the following:

    • before
    • after
    • starting
    • ending
  • title element [0 or 1], with mixed text and (inline prose) element contents

    A title for display and navigation

  • param elements [0 to ∞], with element contents

    Parameters provide a mechanism for the dynamic assignment of value(s) in a control.

  • prop elements [0 to ∞], with text contents

    A value with a name, attributed to the containing control, subcontrol, part, or group.

  • link elements [0 to ∞], with mixed text and (inline prose) element contents

    A reference to a local or remote resource

  • part elements [0 to ∞], with element contents

    A partition or component of a control or part

Address line: addr-line element

An element, with text contents

A single line of an address.

This element appears inside: address.

Address: address element

A postal address.

This element appears inside: person, org.

The address element has the following contents (in order):

  • type attribute (optional), type string

    Indicates the type of address.

    The value may be locally defined, or one of the following:

    • home: A home address.
    • work: A work address.
  • addr-line elements [0 to ∞], with text contents

    A single line of an address.

  • city element [0 or 1], with text contents

    City, town or geographical region for mailing address

  • state element [0 or 1], with text contents

    State, province or analogous geographical region for mailing address

  • postal-code element [0 or 1], with text contents

    Postal or ZIP code for mailing address

  • country element [0 or 1], with text contents

    Country for mailing address

Hash algorithm: algorithm attribute

An attribute, type string

Method by which a hash is derived

This attribute appears on: hash.

Remarks

Valid values MUST be a value defined in the W3C XML Security Algorithm Cross-Reference Digest Methods (W3C, April 2013).

Include all: all element

An element, type empty

Include all controls from the imported resource (catalog)

This element appears inside: include.

The all element may have the attribute:

  • with-child-controls attribute (optional), type string

    The value must be one of the following:

    • yes: Include child controls with an included control.
    • no: When importing a control, only include child controls that are also explicitly called.
Remarks

This element provides an alternative to calling controls individually from a catalog. But this is also the default behavior when no include element is given in an import; so ordinarily one might not see this element unless it is for purposes of including its @with-subcontrols='yes'

Alteration: alter element

An Alter element specifies changes to be made to an included control when a profile is resolved.

This element appears inside: modify.

The alter element has the following contents (in order):

  • control-id attribute (optional), type string
  • remove elements [0 to ∞], type empty

    Specifies elements to be removed from a control, in resolution

  • add elements [0 to ∞], with element contents

    Specifies contents to be added into controls, in resolution

Remarks

Use @control-id to indicate the scope of alteration.

It is an error for two alter elements to apply to the same control. In practice, multiple alterations can be applied (together), but it creates confusion.

At present, no provision is made for altering many controls at once (for example, to systematically remove properties or add global properties); extending this element to match multiple control IDs could provide for this.

As is: as-is element

An element, type boolean

An As-is element indicates that the controls should be structured in resolution as they are structured in their source catalogs. It does not contain any elements or attributes.

This element appears inside: merge.

Back matter: back-matter element

A collection of citations and resource references.

This element appears inside: profile.

The back-matter element has the following contents (in order):

  • citation elements [0 to ∞], with element contents

    A citation to resources, either external or internal (by means of internal cross-reference).

  • resource elements [0 to ∞], with element contents

    A resource associated with the present document.

Remarks

Provides a collection of identified citation and resource objects that can be referenced by a link with a rel value of "reference" and an href value that is a fragment "#" followed by a reference to a citation identifier. A citation can reference a resource by specifying a local target value using the same fragment-identifier approach.

  • The following is a contrived example to show the use of link, citation, and resource.

    <oscal>
            
<link rel="reference" href="#citation1">My citation</link>
            ...snip...
            
<back-matter>
               
<citation id="citation1">
                  
<target>#resource1</target>
               </citation>
               
<resource id="resource1">
                  
<rlink href="https://example.org/some-resource"></rlink>
               </resource>
            </back-matter>
         </oscal>

Call: call element

An element, type empty

Call a control or subcontrol by its ID

This element appears inside: custom, group, include, exclude.

The call element may have attributes:

  • control-id attribute (optional), type string
  • with-child-controls attribute (optional), type string

    The value must be one of the following:

    • yes: Include child controls with an included control.
    • no: When importing a control, only include child controls that are also explicitly called.
Remarks

If @with-subcontrols is yes on the call to a control, no sibling callelements need to be used to call any controls appearing within it (subcontrols. Since generally, this is how control enhancements are represented (as controls within controls), this provides a way to include controls with all their dependent controls (enhancements) without having to call them individually.

Choice: choice element

A value selection among several such options

This element appears inside: select.

Citation: citation element

A citation to resources, either external or internal (by means of internal cross-reference).

This element appears inside: back-matter.

The citation element has the following contents (in order):

  • id attribute (required), type string
  • target elements [0 to ∞], with text contents

    An address for retrieval of a citation

  • title element [0 or 1], with mixed text and (inline prose) element contents

    A title for display and navigation

  • desc element [0 or 1], with text contents

    A short textual description

  • doc-id elements [0 to ∞], with text contents

    A document identifier qualified by an identifier type.

  • Any element (in a foreign namespace)
Remarks

The structure of a citation is not intended to be a full biblographic representation of a cited resource. Instead, it is expected that matching to a source of bibliograpical information will be handled by use of a doc-id. Use of a canonical identifier such as a DOI is recommended for any citation.

Within this element, a generic title can cite a title. If citation information takes the form of a bibliographic entry, then desc can be used to provide it.

City: city element

City, town or geographical region for mailing address

This element appears inside: address.

Class: class attribute

An attribute, type NMTOKENS

Indicating the type or classification of the containing object

This attribute appears on: prop, param, part, set.

Remarks

A class can be used in validation rules to express extra constraints over items of a specific class value. Some objects where this flag appears also permit a name and a ns to be assigned, which can provide for more rigor in validation.

Combination rule: combine element

An element, type empty

A Combine element defines whether and how to combine multiple (competing) versions of the same control

This element appears inside: merge.

The combine element may have the attribute:

  • method attribute (optional), type string

    The value must be one of the following:

    • use-first: Use the first definition - the first control with a given ID is used; subsequent ones are discarded
    • merge: Merge - controls with the same ID are combined
    • keep: Keep - controls with the same ID are kept, retaining the clash
Remarks

Whenever combining controls from multiple (import) pathways, an issue arises of what to do with clashing invocations (multiple competing versions of a control).

This setting permits a profile designer to apply a rule for the resolution of such cases. In a well-designed profile, such collisions would ordinarily be avoided, but this setting can be useful for defining what to do when it occurs.

Custom grouping: custom element

A Custom element frames a structure for embedding represented controls in resolution.

This element appears inside: merge.

The custom element has the following contents:

  • A choice between:
    • group elements [0 to ∞], with element contents

      As in catalogs, a group of (selected) controls or of groups of controls

    • call elements [0 to ∞], type empty

      Call a control or subcontrol by its ID

    • match elements [0 to ∞], type empty

      Select controls by (regular expression) match on ID

Remarks

The custom element represents a custom arrangement or organization of controls in the resolution of a catalog.

While the as-is element provides for a restitution of a control set's organization (in one or more source catalogs), this element permits the definition of an entirely different structure.

Document Identifier: doc-id element

An element, with text contents

A document identifier qualified by an identifier type.

This element appears inside: metadata, citation.

The doc-id element must have the attribute:

  • type attribute (required), type string

    Qualifies the kind of document identifier.

    The value may be locally defined, or one of the following:

    • doi: A Digital Object Identifier (DOI); use is preferred, since this allows for retrieval of a full bibliographic record.
Remarks

A document identifier or name, qualified by a type that indicates the naming system of the identifier. A document identifier may provide a hint to the persistant location of the same or another version of the document.

Specific usage rules, nomenclatures, and recognized file types (e.g., DOIs, CMS locators, etc.) may be defined at application level. The value of this field may also be a hash or checksum.

Exclude controls: exclude element

Which controls to exclude from the resource (source catalog) being imported

This element appears inside: import.

The exclude element has the following contents:

  • A choice between:
    • call elements [0 to ∞], type empty

      Call a control or subcontrol by its ID

    • match elements [0 to ∞], type empty

      Select controls by (regular expression) match on ID

Remarks

Within exclude, all is not an option since it makes no sense. However, it also makes no sense to use exclude/call except with include/all; you would not want to include and exclude something by ID simultaneously. If this happens, an error condition will be reported.

Hash: hash element

An element, with text contents

A representation of a cryptographic digest generated over a resource using a hash algorithm.

This element appears inside: rlink.

The hash element must have the attribute:

  • algorithm attribute (required), type string

    The value may be locally defined, or one of the following:

    • SHA-224
    • SHA-256
    • SHA-384
    • SHA-512
    • RIPEMD-160
Remarks

A hash value can be used to authenticate that a referenced resource is the same resources as was pointed to by the author of the reference.

When appearing as part of a resource (data object), the hash should apply to the referenced object.

An attribute, type string

When selecting, a requirement such as one or more

This attribute appears on: select.

hypertext reference: href attribute

An attribute, type uri-reference

A link to a document or document fragment (actual, nominal or projected)

This attribute appears on: link, rlink, import.

Remarks

Note in particular that href targets include elements (locations) in representations of documents that exist only by virtue of dynamic application, such as (for example) the results of profile resolution.

Identifier: id attribute

An attribute, type ID

Unique identifier of the containing object

This attribute appears on: prop, party, resource, role, citation, param, usage, part, profile.

Remarks

No mechanism is proposed to ensure that id values do not collide across different catalogs. Use profiling without merge to detect such clashes.

Import resource: import element

An Import element designates a catalog, profile, or other resource to be included (referenced and potentially modified) by this profile.

This element appears inside: profile.

The import element has the following contents (in order):

  • href attribute (required), type string
  • include element [0 or 1], with element contents

    Specifies which controls to include from the resource (source catalog) being imported

  • exclude element [0 or 1], with element contents

    Which controls to exclude from the resource (source catalog) being imported

Remarks

An import indicates a source whose controls are to be included (referenced and modified) in a profile. This source will either be a catalog whose controls are given (by value), or a profile with its own control imports (with possible settings.

The contents of the import element indicate which controls from the source will be included. Controls may be either selected (using an include element) or de-selected (using an exclude element) from the source catalog or profile.

When no include is given (whether an exclude is given or not), an include/all is assumed (that is, all controls will be included by default).

  • <import href="catalog.xml">
<include>
<call control-id="ac-1"></call></include></import>

Include controls: include element

Specifies which controls to include from the resource (source catalog) being imported

This element appears inside: import.

The include element has the following contents:

  • A choice between:
    • all element [0 or 1], type empty

      Include all controls from the imported resource (catalog)

    • call elements [0 to ∞], type empty

      Call a control or subcontrol by its ID

    • match elements [0 to ∞], type empty

      Select controls by (regular expression) match on ID

Remarks

To be schema-valid, this element must contain either (but not both) a single all element, or a sequence of call elements.

If this element is not given, it is assumed to be present with contents all (qv); i.e., all controls are included by default, unless the include instruction calls controls specifically.

Parameter label: label element

An element, with mixed text and (inline prose) element contents

A placeholder for a missing value, in display.

This element appears inside: param, set.

Last modified date: last-modified-date element

An element, type dateTime-with-timezone

Date of last modification.

This element appears inside: metadata.

Remarks

This value represents the point in time when the OSCAL document was last updated. The date value must be an ISO 8601 formatted date with full time and time zone indicated. Ordinarily, a date value will be machine generated at time of creation or modification.

In some cases, an OSCAL document may be derived from some source material in a different format. In such a case, the last-modified-date value should indicate the modification time of the OSCAL document, not the source material.

Link: link element

An element, with mixed text and (inline prose) element contents

A reference to a local or remote resource

This element appears inside: metadata, param, part, set, add.

The link element may have attributes:

  • href attribute (required), type string
  • rel attribute (optional), type string
  • media-type attribute (optional), type string
Remarks

This is a rough analog to an HTML anchor (a), except this is a line-oriented (block) element.

The href must point to a local or remote resource. If a local resource is used, this will be indicated by a fragment "#" followed by an identifier which references an identified citation or resource in the document's back-matter. If a remote resource is used, the href value will be an absolute or relative URI pointing to the location of the referenced resource. A relative URI can be resolved relative to the location of the document conatining the link.

To provide a cryptographic hash for a remote target resource, a local reference to a resource is needed. The resource allows one or more hash values to be provided using the rlink hash object.

The media-type provides a hint about the content model of the referenced resource. A valid entry from the IANA Media Types registry

  • The following is a contrived example to show the use of link, citation, and resource.

    <oscal>
            
<link rel="reference" href="#resource1">My Hashed Resource</link>
        ...snip...
        
<back-matter>
               
<resource id="resource1">
                  
<rlink href="https://example.org/some-resource">
                     
<hash algorithm="sha512">C2E9C1..snip..F88D2E</hash>
                  </rlink>
               </resource>
            </back-matter>
         </oscal>

Merge controls: merge element

A Merge element merges controls in resolution.

This element appears inside: profile.

The merge element has the following contents (in order):

  • combine element [0 or 1], type empty

    A Combine element defines whether and how to combine multiple (competing) versions of the same control

  • A choice between:
    • as-is element [0 or 1], type boolean

      An As-is element indicates that the controls should be structured in resolution as they are structured in their source catalogs. It does not contain any elements or attributes.

    • custom element [0 or 1], with element contents

      A Custom element frames a structure for embedding represented controls in resolution.

Remarks

The contents of the merge element may be used to reorder or restructure controls by indicating an order and/or structure in resolution.

Implicitly, a merge element is also a filter: controls that are included in a profile, but not included (implicitly or explicitly) in the scope of a merge element, will not be merged into (will be dropped) in the resulting resolution.

Name: name attribute

An attribute, type string

Identifying the purpose and intended use of the property, part or other object.

This attribute appears on: prop, part.

Remarks

This flag is intended to be used as a control point for validation of properties and parts with respect to constraints related to local semantics. A name is not necessarily expected to be unique, even within the scope of the parent (containing) structure. In some local applications, it is possible for a name to be unique and uniqueness of a name can be validated in such cases.

The definition of a name is subject to its namespace, offered by an ns flag (q.v.). When ns is not given, the namespace is assumed to be oscal and the name should be one of the recognized values for names in OSCAL.

Parameter: param element

Parameters provide a mechanism for the dynamic assignment of value(s) in a control.

This element appears inside: add.

The param element has the following contents (in order):

  • id attribute (required), type string
  • class attribute (optional), type string
  • depends-on attribute (optional), type string
  • label element [0 or 1], with mixed text and (inline prose) element contents

    A short name for the parameter.

    The label value should be suitable for inline display in a rendered catalog.

  • usage elements [0 to ∞], with mixed text and (inline prose) element contents

    Indicates and explains the purpose and use of a parameter

  • constraint elements [0 to ∞], with text contents

    A formal or informal expression of a constraint or test

  • guideline elements [0 to ∞], with element contents

    A prose statement that provides a recommendation for the use of a parameter.

  • A choice between:
    • value element [0 or 1], with mixed text and (inline prose) element contents

      A recommended parameter value or set of values.

      A value provided in a catalog can be redefined at any higher layer of OSCAL (e.g., Profile).

    • select element [0 or 1], with element contents

      A set of parameter value choices, that may be picked from to set the parameter value.

      .

  • link elements [0 to ∞], with mixed text and (inline prose) element contents

    A reference to a local or remote resource

  • Any element (in a foreign namespace)
Remarks

In a catalog, a parameter is typically used as a placeholder for the future assignment of a parameter value, although the OSCAL model allows for the direct assignment of a value if desired by the control author. The value may be optionally used to specify one or more values. If no value is provided, then it is expected that the value will be provided at the Profile or Implementation layer.

A parameter can include a variety of metadata options that support the future solicitation of one or more values. A label provides a textual placeholder that can be used in a tool to solicit parameter value input, or to display in catalog documentation. The desc provides a short description of what the parameter is used for, which can be used in tooling to help a user understand how to use the parameter. A constraint can be used to provide criteria for the allowed values. A guideline provides a recommendation for the use of a parameter.

Part: part element

A partition or component of a control or part

This element appears inside: part, set, add.

The part element has the following contents (in order):

  • id attribute (optional), type string

  • ns attribute (optional), type string
  • class attribute (optional), type string
  • title element [0 or 1], with mixed text and (inline prose) element contents

    A title for display and navigation

  • prop elements [0 to ∞], with text contents

    A value with a name, attributed to the containing control, subcontrol, part, or group.

  • prose element [0 or 1], as prose content: paragraphs, lists etc.

    Prose permits multiple paragraphs, lists, tables etc.

  • part elements [0 to ∞], with element contents

    A partition or component of a control or part

  • link elements [0 to ∞], with mixed text and (inline prose) element contents

    A reference to a local or remote resource

  • Any element (in a foreign namespace)
Remarks

A part provides for logical partitioning of prose, and can be thought of as a grouping structure (e.g., section). A part can have child parts allowing for arbitrary nesting of prose content (e.g., statement hierarchy). A part can contain prop objects that allow for enriching prose text with structured name/value information.

A part can be assigned an optional id>, which allows for internal and external references to the textual concept contained within a part. A id provides a means for an OSCAL profile, or a higher layer OSCAL model to reference a specific part within a catalog. For example, an id can be used to reference or to make modifications to a control statement in a profile.

Use of part and prop provides for a wide degree of extensibility within the OSCAL catalog model. The optional ns provides a means to qualify a part's name, allowing for organization-specific vocabularies to be defined with clear semantics. Any organization that extends OSCAL in this way should consistently assign a ns value that represents the organization, making a given namespace qualified name unique to that organization. This allows the combination of ns and name to always be unique and unambiguous, even when mixed with extensions from other organizations. Each organization is responsible for governance of their own extensions, and is strongly encouraged to publish their extensions as standards to their user community. If no ns is provided, the name is expected to be in the "OSCAL" namespace.

To ensure a ns is unique to an organization and naming conflicts are avoided, a URI containing a DNS or other globally defined organization name should be used. For example, if FedRAMP and DoD both extend OSCAL, FedRAMP will use the ns "https://fedramp.gov", while DoD will use the ns "https://defense.gov" for any organization specific name.

Tools that process OSCAL content are not required to interpret unrecognized OSCAL extensions; however, OSCAL-compliant tools should not modify or remove unrecognized extensions, unless there is a compelling reason to do so, such as data sensitivity.

  • <part name="statement" id="statement-A">
<part ns="https://fedramp.gov" name="status" id="statement-A-FedRAMP">Something FedRAMP Cares About</part>
<part ns="https://defense.gov" name="status" id="statement-A-DoD">Something DoD Cares About</part></part>

Profile: profile element

profile is the root (containing) element of this schema.

Each OSCAL profile is defined by a Profile element

The profile element has the following contents (in order):

  • id attribute (required), type string
  • metadata element [1], with element contents

    Provides information about the publication and availability of the containing document.

  • import elements [0 to ∞], with element contents

    An Import element designates a catalog, profile, or other resource to be included (referenced and potentially modified) by this profile.

  • merge element [0 or 1], with element contents

    A Merge element merges controls in resolution.

  • modify element [0 or 1], with element contents

    Set parameters or amend controls in resolution

  • back-matter element [0 or 1], with element contents

    A collection of citations and resource references.

Remarks

An OSCAL document that describes a selection with possible modification of multiple controls from multiple catalogs. It provides mechanisms by which controls may be selected (import), merged or (re)structured (merge), and emended (modify). OSCAL profiles may select subsets of control sets, set parameter values for them in application, and even qualify the representation of controls as given in and by a catalog. They may also serve as sources for further modification in and by other profiles, that import them.

Property: prop element

An element, with text contents

A value with a name, attributed to the containing control, subcontrol, part, or group.

This element appears inside: metadata, part, add.

The prop element may have attributes:

  • name attribute (required), type string

    The value may be locally defined, or one of the following:

    • keywords: Used to provide a listing of keywords. This is typically used on a prop in the metadata to provide keywords for the document.
    • label: A human-readable label for the parent context.
    • method: An assessment method.
    • status: The status of a control or subcontrol. The value can be 'withdrawn' to indicate that the control or subcontrol has been withdrawn and should no longer be used.
  • id attribute (optional), type string
  • ns attribute (optional), type string
  • class attribute (optional), type string
Remarks

Properties permit the deployment and management of arbitrary controlled values, with and among control objects (controls, parts, and extensions), for any purpose useful to an application or implementation of those controls. Typically, properties will be used to sort, select, order, and arrange controls or relate them to one another or to class hierarchies, taxonomies, or external authorities.

The value of a property is a simple scalar value or list of values.

The lexical composition of properties may be constrained by declarations including matching to regular expressions or by declaring a known datatype.

Because properties are often used as selectors or identifiers for OSCAL operations, their values can be expected frequently to be normalized (e.g., with respect to whitespace) in use; however, this is application defined.

For singletons (that is, the only element among siblings with its class), properties are especially useful as proxies (unique identifiers) for their controls, such that controls may be returned one for one on queries for properties (name and value). The robustness of such queries can be ensured by appropriate property declarations (as singletons and as identifiers); cf declare-prop in the declarations model (which also supports other constraints over property values).
